# AHP verified core — C++ distribution (via the Dafny Rust backend)

A native library with a C ABI, produced from the formally-verified Dafny AHP
core, consumable from C++.

```
Dafny core (verified)  ──dafny translate rs──▶  Rust  ──cargo──▶  libahp_core.a / .dylib
                                                                        │
                                                    ahp_core.h  ◀───────┤ C ABI
                                                                        │
                                                            your C++ ───┘
```

**Status: working.** `clang++` compiles a program against `ahp_core.h`, links
`libahp_core.a`, calls the reducers, and prints correct results. The verified
core's own 148-fixture conformance corpus also runs green through this library.

## Trust chain — read this first

This section is the point of the package. Be precise about what is proven.

| Layer | Provenance | Verified? |
|---|---|---|
| AHP reducers (`apply1`), kernel fold, codec, protocol datatypes | Dafny, proven | **Yes** — proofs discharged by Dafny/Z3 |
| Rust for all of the above | machine-generated by `dafny translate rs` | **Derived from proofs.** Not re-proven; trusts the Dafny Rust backend |
| `ConfluxJsonText.SortedKeySetRuntime` rewrite | one required source change | **Yes** — re-verified by Dafny (1623 verified, 0 errors) |
| `ahp-core-ffi/src/ffi.rs` (the C ABI) | **hand-written** | **No** |
| `ahp-core-ffi/src/_dafny_externs.rs` (host capabilities) | **hand-written stubs** | **No** — and deliberately not implemented; see below |
| `include/ahp_core.h` | **hand-written** | **No** |

**In one sentence:** the code that *executes your protocol logic* is machine-derived
from the proven spec; the code that *marshals it across the C boundary* is
ordinary hand-written glue with ordinary bugs.

Nothing here is a hand-port. No reducer was reimplemented in Rust, C, or C++.

### What is NOT covered

- **The binding layer.** Pointer/lifetime/encoding bugs in `ffi.rs` or
  `ahp_core.h` are possible and are not excluded by any proof.
- **The Dafny→Rust backend.** Its correctness is assumed, exactly as the C#, Go,
  Java, JS, and Python extractions in this repo assume theirs.
- **Host I/O capabilities.** Filesystem, process execution, clocks, sockets, and
  HTTP are `{:extern}` in the Dafny core — the *unverified host boundary* by
  design. This build ships **stubs that abort loudly** rather than fabricate
  data. The reducers are pure and never call them. If you need them, implement
  `_dafny_externs.rs`; the signatures are already correct.

## Why this route, and not `dafny translate cpp`

Dafny ships an experimental C++ backend. **It cannot compile this core**, and
that is not a matter of effort — the backend lacks features the core is built
out of.

Verified against the newest Dafny that exists (nightly `4.11.1+368adac`,
2026-07-07), on `spec/version.dfy`, the core's *simplest, dependency-free*
module:

```
$ dafny translate cpp spec/version.dfy
Error: Feature not supported for this compilation target: Unicode chars

$ dafny translate cpp spec/version.dfy --unicode-char:false
Error: Feature not supported for this compilation target: Unbounded integers
Error: Feature not supported for this compilation target: Non-native numeric newtypes
```

The core has 60 `: int` and 47 `nat` fields in its protocol datatypes and 26
arrow-typed (higher-order) signatures in the reducer core. `Docs/Compilation/Cpp.md`
lists exactly these as unsupported. Peeling one blocker only reveals the next.
Full transcript: [`evidence/03-nightly-cpp-and-rs.txt`](evidence/03-nightly-cpp-and-rs.txt).

The Rust backend has none of these limits — it emits `DafnyInt` (arbitrary
precision, backed by `num::BigInt`) and `Rc<dyn Fn>` for function values — and
Rust exposes a C ABI natively. Hence: Dafny → Rust → C ABI → C++.

## The one source change, and why it is still verified

The Rust backend **mandates** `--enforce-determinism` (there is no opt-out; the
nightly still requires it). That flag rejects the *assign-such-that statement*
form, `var k :| P(k);`, unconditionally — even when the witness is provably
unique.

The conflux runtime contains **exactly one** such statement in compiled code:
`ConfluxJsonText.SortedKeySetRuntime`, which picks the lexicographically least
key when canonicalising JSON object ordering. (The other 25 occurrences of `:|`
are inside `ghost` lemmas and functions, which are erased before compilation and
are unaffected.)

`scripts/patch-determinism.py` lifts that statement into a compiled *function*
using the *let-such-that expression* form, which the flag permits, carrying the
**identical predicate**:

```dafny
function LeastKeyOf(s: set<string>): string
  requires s != {}
  requires exists k :: k in s && (forall y :: y in s ==> !StrLt(y, k))
  ensures LeastKeyOf(s) in s
  ensures forall y :: y in s ==> !StrLt(y, LeastKeyOf(s))
{
  assert forall a, b :: ... ==> a == b by { ... LeastKeyUnique(s, a, b); }
  var k :| k in s && (forall y :: y in s ==> !StrLt(y, k));
  k
}
```

Dafny additionally requires a *compiled* let-such-that to be **uniquely
determined**. It is: the runtime's own `LeastKeyUnique` lemma already proves any
two least keys of `s` are equal (via `StrLtTrichotomy`), and the `assert ... by`
discharges that obligation. The selected string is therefore forced to be
`LexMin(s)` — the same value the statement form yielded.

`SortedKeySetRuntime`'s postcondition (`ensures keys == SortedKeySet(s)`) is
unchanged, and **`regenerate.sh` re-verifies the whole patched runtime before
translating**:

```
==> re-verifying patched runtime (this is the proof the rewrite is sound)
Dafny program verifier finished with 1623 verified, 0 errors
```

**No AHP reducer was touched.** The change is in a runtime JSON-canonicalisation
helper, and it is proof-backed, not asserted.

> Why the source at all, rather than the shipped `conflux-runtime.doo`? The
> `.doo` was built with `enforce-determinism = false`, and Dafny refuses to mix:
> `"--enforce-determinism is set locally to True, but the library was built with
> False"`. A `.doo` is a zip whose `program` member *is* the runtime's Dafny
> source, so `regenerate.sh` recovers it and recompiles from source.

## Build

Requires `dafny` 4.11.0, a Rust toolchain, and a C++17 compiler.

```bash
cmake -S . -B build-cmake -DCMAKE_BUILD_TYPE=Release
cmake --build build-cmake
ctest --test-dir build-cmake --output-on-failure
```

CMake drives `cargo` and exposes the result as an ordinary imported target:

```cmake
target_link_libraries(your_app PRIVATE ahp_core)
```

To regenerate everything from the Dafny core — extract runtime source, patch,
**re-verify**, translate, build, and run both C++ programs:

```bash
./regenerate.sh
```

Override the core location with `AHP_CORE=/path/to/agent-host-protocol-core-dafny`.

## Using it

```c
#include "ahp_core.h"

AhpTerminalState *s0 = ahp_terminal_initial();          // Dafny Terminal.T0()
AhpTerminalState *s1 = ahp_terminal_cwd_changed(s0, "/tmp");   // verified apply1
AhpTerminalState *s2 = ahp_terminal_resized(s1, 120, 40);      // verified apply1

char *cwd = ahp_terminal_cwd(s2);                       // "/tmp"
ahp_string_free(cwd);

ahp_terminal_free(s2); ahp_terminal_free(s1); ahp_terminal_free(s0);
```

Transitions are **pure**: they return a new state and never mutate or consume the
input, which stays valid and independently owned. Every returned handle needs
`ahp_terminal_free`; every returned `char*` needs `ahp_string_free`.

`ahp_terminal_cwd` returns `NULL` when the reducer holds `None` — distinct from
an empty string. The `*_size` / `*_exit_code` readouts return `1` when present
and `0` for `None`, so optionality survives the C boundary.

## Evidence

`examples/ahp_terminal_demo.cpp` asserts on expected values taken from the Dafny
core's own corpus (`spec/terminal.dfy` `RunCorpus`) — not values read back off
this implementation — so it fails if reducer behaviour changes. Confirmed
falsifiable: changing one expectation makes it exit 1.

```
$ ./build-cmake/ahp_terminal_demo
initial state (Dafny Terminal.T0()):
  ok   title == "bash"
  ok   cwd is None (NULL, not empty string)
  ok   exitCode is None
  ok   content is empty

single transitions (verified reducer apply1):
  ok   cwdChanged("/tmp")            -> cwd == "/tmp"
  ok   titleChanged("zsh")           -> title == "zsh"
  ok   resized(120, 40)              -> cols == 120, rows == 40
  ok   exited(130)                   -> exitCode == 130

purity (reducer returns a new state; inputs untouched):
  ok   s0 still "bash" after s2 became "zsh"
  ok   s3 exitCode still None after s4 exited(130)

kernel fold (higher-order Terminal.fold over ConfluxContract.Fold):
  ok   fold over 3 data actions returned a state
  ok   fold(s0, [3 actions]) content_len == apply1 x3 content_len (1 == 1)

boundary behaviour:
  ok   NULL state handle             -> NULL
  ok   NULL string argument          -> NULL

ALL CHECKS PASSED -- C++ successfully drove the verified AHP reducer.
```

The kernel-fold check matters: `Fold` takes a *function value*. That is precisely
what the Dafny C++ backend cannot express, and it works here.

## Scope

The C ABI currently surfaces the **Terminal** channel plus the full-corpus
runner. All eight channel reducers (`Root`, `ResourceWatch`, `Canvas`,
`Changeset`, `Annotations`, `Terminal`, `Session`, `Chat`) are present and
working in the compiled library — the corpus exercises every one — so widening
the C ABI is additional binding code in `ffi.rs`, not additional extraction work.
