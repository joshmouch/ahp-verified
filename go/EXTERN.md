# The extern trust boundary

Most of this module is mechanically translated from a formally verified Dafny
specification. Three packages are not, and this file says exactly which and why.

## What is verified

Everything under the translated packages — the seven AHP channel reducers
(`Chat`, `Session`, `Canvas`, `Terminal`, `Changeset`, `Annotations`,
`ResourceWatch`), the convergence and fold proofs, and the JSON codec
(`ConfluxCodec`, `ConfluxJsonText`) — is generated by `dafny translate go` from
the proven source. No human edited that output, with one exception recorded
below.

## What is not verified

Dafny's `{:extern}` declarations mark points where the specification deliberately
stops and the host takes over: clocks, process execution, the filesystem, and
sockets cannot be proven, only trusted. Every language binding of this core ships
hand-written adapters at that boundary — C# in `ConfluxCommandIdentityAdapter.cs`
and friends, JavaScript in `ConfluxIoAdapter.js`, and Go here:

| Package | File | What it supplies |
|---|---|---|
| `ConfluxCommandIdentityCapability` | `adapter.go` | strict UTF-8, NFC normalization, SHA-256 |
| `ConfluxIoCapability` | `adapter.go` | process execution, file reads, monotonic clock, RSS sampling, filesystem watching |
| `ConfluxHttpCapability` | `adapter.go` | response writing and server-sent-event streaming |

These adapters are ordinary Go code. They carry no proof. They were written to
mirror the semantics of the existing C# and JavaScript adapters, and the
`ConfluxCommandIdentityCapability` behavior is pinned by the tests in
`ahp_smoke_test.go` — including a published SHA-256 test vector and negative
cases for non-canonical and invalid-UTF-8 input.

Treat them the way you would treat any unproven dependency: review them.

### Coverage note

`ConfluxSocketCapability` and `ConfluxWebSocketCapability` are declared by the
core but no reachable code path in this translation calls into them, so they have
no Go adapter. If a future version of the core reaches them, the build will fail
loudly with an `undefined: Companion_Default___` error rather than silently
degrading.

`ConfluxIoCapability`'s handle-based process operations (`ReadProcessOutputWithin`,
`ProcessRssMb`, `ProcessExitStatus`, `TerminateProcessTreeWithin`) are backed by a
real registry, but nothing in the reachable Go surface currently calls
`SpawnProcess` to populate it. They are implemented for hosts that drive the
registry directly; they return the documented not-found result otherwise.

`ConfluxHttpCapability` resolves the core's integer request handles through a
registry that a Go host populates by calling `Register` from its own
`http.Handler`. Calls against an unregistered handle are dropped rather than
panicking, matching the C# and JavaScript adapters' handling of stale handles.

## The one edit to generated code

`ConfluxWatermark/ConfluxWatermark.go` required a one-line fix. Dafny's Go backend
emitted:

```go
func (CompanionStruct_Cap_) Create_Cap_(Pos _dafny.Int, Cap _dafny.Int) Cap {
	return Cap{Cap_Cap{Pos, Cap}}
}
```

The parameter `Cap` shadows the type `Cap`, so `return Cap{...}` resolves to the
parameter and the package does not compile (`Cap (parameter) is not a type`). The
parameter is renamed to `Cap_`. This is a rename of a function-local binding; it
does not change behavior. `regenerate.sh` reapplies it automatically, and this is
an upstream Dafny code-generation bug rather than a defect in the specification.
